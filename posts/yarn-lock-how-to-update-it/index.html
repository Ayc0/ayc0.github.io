Let‚Äôs say you depend on a module A that itself depends on B with the version ^1.1.1. The behaviour is indeed intended: the main goal of lock file is to ensure that your dependencies won‚Äôt change if you don‚Äôt ask for it. Real world example: you are using create-react-app, and you want to also to use xo, as both come with their own version of ESLint pre-installed, you could end up with 2 ESLint installed.

This will force all versions of B to resolve to this one version 1.1.5.

I don‚Äôt like this approach, as it‚Äôll force all versions (even incompatible ones like a potential v2 that could also be used in your deps) to be updated to this unique one. So I‚Äôd reserve this for modules you know cannot exist in multiple versions, but I wouldn‚Äôt use it otherwise.

Removing the yarn.lock file

Section titled ‚ÄúRemoving the yarn.lock file‚Äù

If you remove the lock file completely and then run yarn install, yarn will re-resolve all versions to the latest allowed by their specified ranges, and thus fix all those duplicated deps.

I don‚Äôt recommend doing that as not all packages respect the semver convention. So you could introduce a lot of regressions in your code.

Section titled ‚Äúyarn dedupe (recommended)‚Äù

If you‚Äôre using yarn 2+, you have access to the command yarn dedupe B (to dedupe all the B packages).

If you‚Äôre running on yarn 2+, I strongly recommend using this, as it‚Äôs the easiest and safest method of all of those listed here.

Yarn 2+

Section titled ‚ÄúYarn 2+‚Äù

Yarn berry (version 2 and above), has the command yarn why (it already existed in yarn 1, but the output was completely different).

The main difference between yarn list in yarn 1 and yarn why in yarn 2+ is that, in addition to having the version installed + the parent that required this package, you also have the version ranges that were requested each time.
